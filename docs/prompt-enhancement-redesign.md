# Prompt Enhancer 核心优化逻辑重构讨论文档

## 项目背景

Prompt Enhancer 是一款 Chrome 浏览器扩展，功能是一键润色/优化用户输入给 AI 的 prompt。用户在任意 AI 聊天网站（ChatGPT、Gemini、DeepSeek 等）的输入框中写好 prompt 后，点击插件按钮，插件会调用 LLM API 对 prompt 进行优化重写，然后替换回输入框。

## 当前架构

核心流程：用户输入 → Analyzer 分析 → Prompt Builder 构建系统提示 → 调用 LLM → 输出优化后的 prompt

### Analyzer（分析器）
- 通过关键词子串匹配将用户输入分类为 8 种任务类型：CODE / WRITING / ANALYSIS / EXTRACTION / QA / PLANNING / RESEARCH / CHAT
- 通过复杂度信号词检测确定 3 级推理模式：SIMPLE / DEEP_THINKING / EXPERT
- 额外检测：语言（中/英）、是否包含代码片段、是否有格式要求、是否有多个子问题

### Prompt Builder（构建器）
生成发给 LLM 的系统提示，固定结构为：
1. 角色设定（固定为"Prompt Optimization Expert"）
2. 任务描述（固定为"生成结构化的优化版本"）
3. 约束条件（保持语言、保持意图等通用约束）
4. 根据推理模式添加优化级别描述（Light / Deep / Expert）
5. 根据任务类型添加专项优化要点
6. 输出格式指导（固定五段式：角色设定→任务描述→上下文→约束条件→输出格式）

### 实际效果
无论用户输入什么，最终优化出的 prompt 几乎都遵循同一个五段式模板。

## 现有问题诊断

### 问题 1：模板僵化——"万能公式"综合症

不管用户输入的是什么类型、什么长度、什么场景的 prompt，Builder 的输出结构完全一致（角色→任务→上下文→约束→格式）。

具体表现：
- 简单指令（如"帮我翻译这段话"）被过度包装成结构化的长 prompt
- 已经写得很好的 prompt 被彻底重写，原有风格丢失
- 多轮对话中的追问（如"第三点展开讲讲"）被套上不必要的角色设定和背景描述
- 补充说明（如"加上错误处理"）被当成全新的 prompt 处理

Analyzer 虽然做了分类，但分类结果只影响系统提示中的几行指导文字，实际优化模板没有根本差异。

### 问题 2：零上下文感知——永远冷启动

每次优化都是孤立的单次操作，不了解：
- 用户当前在与 AI 进行什么主题的对话
- 这是第几轮对话（首轮 vs 追问）
- 前面几轮讨论了什么内容
- 用户是否在持续围绕同一个项目/任务工作

后果：
- 对话中间轮次的追问被加上冗余的角色设定和背景描述（AI 已经知道上下文了）
- 无法识别"这是在修正上一轮的方向"vs"这是全新的请求"
- 同一项目中的连续 prompt 之间没有一致性

### 问题 3：Analyzer 关键词检测过于粗糙

子串匹配的误判率高。例如：
- "写一个 Python 爬虫"同时命中 CODE（"python"）和 WRITING（"写"）
- 分类结果取决于关键词列表的长度而非语义理解
- 短文本信号不足时频繁退回到 CHAT 兜底类型

## 讨论方向

### 方向 1：动态优化策略（替代固定模板）

核心思路：根据输入的实际特征选择不同的优化策略，而不是始终套用同一模板。

初步设想的策略分层：

| 输入特征 | 优化策略 | 具体做法 |
|---------|---------|---------|
| 简短指令（<30字，意图明确） | 轻润色 | 只补充缺失的关键信息（如目标语言、格式要求），保持简洁 |
| 首轮复杂请求（长文本，新话题） | 结构化重写 | 完整的框架搭建：角色+任务+约束+输出规范 |
| 多轮对话中的追问 | 意图澄清 | 让追问更具体、减少歧义、引用上文关键点 |
| 已经写得不错的 prompt | 微调锐化 | 保留原有结构，只做措辞锐化和遗漏补充 |
| 补充/修正类指令 | 约束追加 | 保持简短，明确新增或修改的约束 |

需要讨论：
- 如何准确判断输入属于哪种场景？
- 每种策略的 system prompt 应该如何设计？
- 是否有更好的策略分类方式？

### 方向 2：会话上下文记忆

核心思路：插件记住同一会话中的历史输入，用于辅助优化决策。

初步设想的分阶段方案：

**第一阶段（轻量级）**：
- 在同一页面的 content script 中维护一个滑动窗口，记录最近 3-5 轮用户发出的 prompt
- 用于判断：当前输入是"新话题"还是"追问/补充"
- 如果是追问，优化策略自动切换为"意图澄清"模式，不再套结构化模板
- 页面关闭或刷新即清空，无持久化

**第二阶段（进阶，暂不实现）**：
- 持久化会话摘要到 chrome.storage
- 跨标签页/跨会话的项目上下文记忆
- 需要设计 UI 让用户管理记忆（查看、清除、切换项目）

需要讨论：
- 如何判断"新话题"vs"追问"？基于时间间隔？基于语义相似度？还是基于页面 URL？
- 历史 prompt 发送给 LLM 时如何组织？全量发送还是摘要？
- 记忆窗口的大小如何确定？是否需要用户可配置？
- 是否需要把 AI 的回复也纳入上下文？（技术上可行但需要从页面提取）

### 方向 3：让 LLM 自己判断策略（可选讨论）

另一种思路是：不在客户端做复杂的规则判断，而是在 system prompt 中让 LLM 自己决定优化策略。

例如：将当前的 Analyzer + Builder 替换为一个更智能的 system prompt，告诉 LLM："先分析这个 prompt 的类型和场景，然后选择合适的优化方式"。这样把分类和策略选择都交给 LLM。

优点：不依赖脆弱的关键词匹配，LLM 的语义理解能力远超规则引擎。
缺点：增加 token 消耗，可能增加延迟，输出稳定性需要验证。

## 当前代码关键文件

- `src/background/analyzer.ts` — 关键词分析器（任务分类 + 推理模式判定）
- `src/background/prompt-builder.ts` — 系统提示词构建（固定模板）
- `src/background/enhancer.ts` — 协调层（分析 → 构建 → 调用 API）
- `src/shared/types.ts` — 类型定义（TaskType / ReasoningMode / PromptAnalysis）
- `src/shared/constants.ts` — 关键词列表（TASK_INDICATORS / COMPLEXITY_SIGNALS）

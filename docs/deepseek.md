1. 可行性评估与优先级建议
方向1：动态优化策略（替代固定模板）
可行性：高。当前代码已有 Analyzer 和 PromptBuilder，只需扩展分析维度，并为不同场景设计多个 system prompt 模板。改动范围可控，无需大规模重构。

优先级：最高。这是解决“万能公式”综合症最直接有效的手段，能快速提升用户体验。

方向2：会话上下文记忆
可行性：中。在 content script 中维护滑动窗口记录用户输入可行；提取 AI 回复需解析 DOM，复杂度较高且依赖具体网站结构。可先实现轻量级版本（仅记忆用户输入），后续逐步增强。

优先级：中高。多轮对话场景常见，记忆能显著改善追问类 prompt 的优化质量，值得投入。

方向3：让 LLM 自己判断策略
可行性：中低。虽然 LLM 理解能力强，但将策略选择完全交给 LLM 会增加 token 消耗、延迟和不确定性（输出可能偏离预期）。需要精心设计 system prompt 并经过大量测试才能稳定。

优先级：低。可作为长期探索方向，或作为备用方案在规则引擎失效时降级使用。当前阶段优先完善规则+模板的方式更可控。

综合建议：优先实施方向1，同步规划方向2的轻量级实现，方向3暂缓或作为实验性功能。

2. 动态优化策略的具体设计
2.1 输入特征分析（增强 Analyzer）
在现有分类基础上，增加以下可计算特征：

文本长度：短（<30字）、中（30-200字）、长（>200字）。

是否已有结构：检测是否包含“角色：”、“任务：”、“约束：”等关键词。

是否多轮追问：若启用会话记忆，判断当前输入与历史最后一条的关系（如以“然后”、“具体”、“还有”开头，或语义相似度低）。

指令类型：补充/修正（如包含“改为”、“加上”、“不要”）、新请求。

语言风格：简洁/详细（通过句子长度、副词密度粗略判断）。

2.2 优化策略分类及对应 system prompt 设计

2.3 策略选择逻辑
伪代码示例：

typescript
function selectStrategy(analysis: Analysis, history: History): Strategy {
  if (isCorrectionOrAddition(analysis)) return 'CONSTRAINT_APPEND';
  if (hasHistory(history) && isFollowUp(analysis, history)) return 'INTENT_CLARIFY';
  if (isShortAndSimple(analysis)) return 'LIGHT_REFINE';
  if (hasGoodStructure(analysis)) return 'SHARPEN';
  return 'FULL_RESTRUCTURE';
}

3. 会话记忆的技术方案选型（轻量级）
3.1 存储方案
内存数组：在 content script 中维护一个数组 historyPrompts，仅存储当前会话（同一标签页）最近 3-5 条用户输入。

持久化：使用 chrome.storage.session（Chrome 102+）实现会话级持久化，页面刷新不丢失，标签页关闭即清除。若需支持旧版浏览器，可降级至内存。

3.2 会话边界检测
自动重置条件：

页面 URL 变化（如从 chat.openai.com 跳转到 gemini.google.com）

用户输入间隔超过预设阈值（如 30 分钟）

用户主动点击插件“新对话”按钮（可选）

追问判断：基于以下规则（可组合）：

文本相似度：当前输入与最后一条历史输入的编辑距离或 Jaccard 相似度低于阈值（如 0.3）时视为新话题。

关键词触发：以“然后”、“还有”、“具体”、“那”等开头视为追问。

简单起见，初期可仅使用时间间隔 + 开头词规则。

4. 遗漏问题与更好思路
4.1 用户偏好配置
允许用户选择优化风格：

详细程度：简洁/标准/详细

语气风格：专业/友好/中立

是否保留原始 prompt 中的特定短语

可在插件弹出页增加开关或下拉菜单，将偏好传入 system prompt。

4.2 优化结果缓存
对于完全相同的输入，可缓存上次优化结果，避免重复 API 调用。考虑使用 Map 或 chrome.storage.local 设置过期时间。

4.3 错误回退机制
当 LLM 调用失败（网络、API 限流）或输出异常（空、乱码）时，直接返回原始输入，并提示用户。

4.4 混合分类降成本
对于复杂场景，可先用轻量模型（如免费 API 或本地规则）做预分类，再决定是否调用主模型。例如：若用户输入是“你好”，直接返回无需优化，节省 token。

4.5 支持多种 LLM 提供商
目前可能只支持单一 API，可抽象接口以支持切换 OpenAI、Claude、本地模型等，提升容错和灵活性。

总结
方向	可行性	优先级	建议行动
动态优化策略	高	最高	立即开始，基于现有 Analyzer 扩展，设计 5 种策略模板。
会话上下文记忆	中	中高	并行开发，先实现内存滑动窗口 + 简单边界检测。
LLM 自主判断策略	中低	低	作为实验性分支，待前两个方向稳定后探索。
同时，建议引入用户偏好、缓存和错误处理等辅助功能，全面提升体验。以上方案可在保持扩展轻量的前提下，显著改善优化质量。

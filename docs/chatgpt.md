Prompt Enhancer 核心优化逻辑重构报告
1. 现有Prompt优化工具与研究模式
目前市面上已有多款AI提示优化扩展，如Prompt Genie、Promptimize等，它们纷纷引入了上下文记忆和自适应模板等功能。例如Prompt Genie提供“Context Memory”功能：只需一次性输入项目背景，每次优化都会自动复用这段背景
；Promptimize允许用户创建“动态变量”以保持多次提示间的一致性
。从学术角度看，最新研究也强调了动态Prompt构造的重要性：如EmergentMind总结，自适应提示策略能够根据输入上下文、任务特征和反馈动态调整Prompt设计
，显著提升LLM性能；微软提出的动态推荐框架则通过检索增强的领域知识库和层次化技能组织，动态选择相关模板并结合少量示例生成高相关度提示
。此外，Anthropic将“上下文工程”提升到新的高度，指出构建AI系统时不仅要关注单条提示文本的编写，更要规划和维护整个对话的上下文状态
。总之，现有工具和研究趋势表明：1）Prompt优化应考虑对话历史和上下文记忆，2）应采用动态可变的提示模板，而不是简单固定五段结构。

2. 现有实现问题诊断
模板过于死板：目前的Builder总是输出相同结构，导致简单指令也被过度包装，而原本良好的提示被全盘重写。这与EmergentMind所说的静态模板局限相符：静态Prompt容易忽略输入的多样性和重要语义
。
缺乏上下文记忆：每次优化都是冷启动，没有利用历史对话或项目背景。Anthropic指出，随着上下文变长，模型注意力有限，如果不加记忆管理，就会出现“上下文稀释”（context rot）问题
。因而跟多轮对话无关地重新设置角色、背景是效率低下的。
分类粗糙易出错：现用的关键词子串匹配忽略语义，导致分类结果不稳定，比如“写一个Python爬虫”同时匹配了“写”(写作类)和“Python”(编程类)。这种方法依赖预置词表，遇到复杂或跨领域输入时易跌回兜底类型，导致优化策略缺乏针对性。
3. 改进方向比较
3.1 动态优化策略设计
核心思路是根据提示的特征选择不同的优化方式，而非一刀切。可考虑以下策略划分（示例）：

轻量润色（短提示）：对简单、明确的指令（如小于30字的请求）仅补充关键信息、修正措辞，保持简洁风格，避免生成冗余结构。
中类似Claude插件对短提示的“绕过逻辑”示例可参考。
结构化重写（首轮长文）：对首次出现的新话题或复杂任务，采用丰富的结构化输出：明确角色、任务、约束和输出格式，引导模具化思考。相比原模板可适当精简，加入领域知识或示例。
意图澄清（多轮追问）：当检测到对话中的追问或补充时，重点使问题更具体、减少歧义，引用上文要点，而不重复先前已知背景。
所示的OpenMemory，将历史记忆注入当前提示，即是一种思路；我们可以在模板中仅强调新增条件，并自然承接前文上下文。
精调锐化（已好提示）：对于本身结构合理、意图明确的提示，仅在措辞、语气或细节上微调，不改变原有层次；保持原有简洁和风格。
约束补充（修正类）：对于用户明确提出“加xxx条件”之类的提示，在原提示基础上以附加形式增加新约束，避免完全重写，直接明确修改点即可。
为实现上述策略，需要解决的关键问题包括：**如何准确判别输入场景？**建议结合多种方法：例如利用提示长度、语义相似度、关键词模式（如包含“翻译”“摘要”等）、甚至调用LLM进行初步意图分析来综合判断。同时，可对现有Analyzer做逐步替换：比如使用小模型或规则+模型混合分类，而非单纯关键词。对于每种策略，需要单独设计System Prompt模板。例如“轻量润色”可设置
中提到的短提示跳过机制；“结构化重写”可保留类似原固定框架；“意图澄清”则更注重提问的指向性和明确性。在设计时，要考虑EmergentMind所说的自适应提示方法：模板应该尽可能精简但充分，并可根据实例反馈迭代
。

3.2 会话上下文记忆方案
为了避免每次都完全重置上下文，我们应引入简单的记忆机制。阶段一：页面内滚动窗口——在当前标签页的content script中维护一个最近N轮用户提示的缓存（例如最近3~5条），不跨页面保存。每次新的提示到来时，先通过时间间隔和语义相似度判断其与上一条的关系：若判断为“追问/补充”，则触发“意图澄清”策略，不再重复角色和背景；否则视为新话题。OpenMemory扩展所做的正是这件事：它在不同AI界面后端注入历史记忆
。在实现上，可以将历史提示或其摘要作为context提供给Builder，或要求LLM在生成新prompt时考虑“以下是上下文”。

阶段二：持久化记忆（可选）——将会话摘要或重要信息持久化到chrome.storage，以跨标签页/跨会话保持一致性，并提供给用户管理（查看、清除等）。这需要评估：如果长期存储会话，必须处理隐私（仅本地保存，不上传）和存储成本。此外，加入过多上下文可能引发前文“上下文稀释”问题
。常用做法是摘要压缩：用LLM对近期对话生成简短记忆片段
（如自我问答或四元组式记录），并在后续提示中附加此摘要。这样既保留关键信息，又控制token数。我们也可仅在用户请求时回看记忆，以减小实时负担。窗口大小和记忆粒度需实验调优，可考虑让用户配置，例如启用“简要记忆”还是“详细记忆”。此外，可探索将LLM回复也纳入分析，但这需要从页面解析回应并非必要。

隐私/性能权衡：所有内存操作均在客户端完成，不与外部服务共享数据保证隐私。存储历史会增加API调用负担（传递更多tokens），可能略微提高延迟；对话摘要可限制在较短长度以平衡。必须避免存储敏感数据或非必要个人信息。如果用户不需要跨会话记忆，第一阶段的轻量滑窗即可满足需求。

3.3 LLM自主决策策略（Meta-Prompt）
另一种思路是不在插件端写死分类规则，而是构造一个更智慧的System Prompt，让LLM在接收用户输入后先自我判断处理策略。例如可以提示：“首先分析此请求的类型和场景，然后决定是进行轻度润色、结构化重写，还是只修改特定约束，最后输出优化后的提示。”这样做可利用LLM强大的语义理解能力，避免粗糙规则的误判。优点是灵活性高，不需手动维护分类表；缺点是每次要多一个分析步骤，增加token数和延时，且输出稳定性需要严格测试。相关的研究（如EmergentMind讨论的元模型评分）也表明，用模型选择最佳Prompt模板是可行的
。我们可视情况将此策略作为备选或辅助方案，在实际应用中对比规则判断和模型判断的效果。

4. 综合建议与实施计划
方向可行性评估与优先级：建议优先实施动态策略优化，因为它直接改善现有输出质量、易于测试；其次是会话记忆功能，可显著提升多轮对话的连贯性；最后考虑引入LLM自主决策（需要更多尝试和资源）。初期可保持现有API调用不变，仅调整前置逻辑和Prompt模板。

动态策略的实现：扩展Analyzer，让其输出更加细粒度的分类信号（如：简短命令、新话题、多轮追问、修正类等）。为每种情境编写不同的System Prompt模板，避免统一“五段式”结构。比如对“短命令”模板指令为“简短回复，若有遗漏信息请补充”；对“追问”模板指令为“针对以下追问明确回答上一轮对话的细节点”；对“已良好提示”模板指令为“仅微调措辞，保持格式”。可通过实验确定阈值（长度、关键词数量等）并借鉴
中的“跳过阈值”机制。

会话记忆技术路线：先实现本地滑动窗口：在内容脚本中记录最近若干轮用户输入，并在构造系统提示时将其简单拼接或摘要加入上下文提示中（或传入LLM作为“背景”）。使用开源文本相似度算法或固定时限来识别“追问”。后续若进展顺利，可扩展至持久化存储：定期让LLM生成对话摘要并保存（参考递归摘要
的思路），在跨会话时自动加载。性能与隐私：所有处理均在本地进行，不泄露用户数据；摘要控制在几十字，避免大段历史导致响应变慢。用户可配置记忆开关和窗口大小。

遗漏点与其他优化：可以考虑增加提示评估功能，如让LLM给出优化前后提示的评分，以辅助迭代规则；或者集成提示库让用户保存喜欢的优化模板；对于多语言支持，确保保留用户输入语言。调研发现，一些工具还提供协作和版本管理功能，这些可以视产品定位逐步加上。

测试用例设计：制定一系列测试来验证重构效果。例如：

短命令测试：输入极短明确指令（如“翻译这段文本”），检查是否只做轻度润色，而不是生成多余背景；
首轮复杂请求：输入长篇复杂任务（如论文写作大纲需求），检查是否产生全新的结构化提示且包含有效指引；
多轮追问：模拟连续对话场景，后一条为追问（如“前一点再展开一下”），确保新优化提示不重复先前背景，只回答新增需求；
已有优质提示：输入格式良好内容（如已完整的SQL查询描述），检查优化结果应仅微调语句，而不破坏原结构；
补充约束：输入带修正指令（如“在上次提示基础上增加异常处理”），验证输出在原提示基础上正确附加新内容。
性能测试：对比重构前后响应延迟和Token数，确保性能可接受；
错误处理：输入模糊或无意义指令，观察系统是否稳健返回原输入或提示错误。
通过以上方案，我们可以有针对性地重构Prompt Enhancer，提高其灵活性和用户体验，并通过详细测试确保重构结果符合预期
。

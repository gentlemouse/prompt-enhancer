# 输入框右侧图标定位问题调研与整改报告

## 1. 背景与目标

目标是让插件图标在各类 AI 页面中持续稳定地“贴附”输入框右侧，并且不遮挡站点原生按钮，不在滚动/流式生成/布局变化时漂移。

## 2. 复现问题与根因归类

### A. 图标停在页面中间，不跟随输入框
- 现象：输入后页面高度变化或滚动时，图标停留在旧位置。
- 根因：`showButton` 里使用“输入框 ID 缓存”避免重复定位，导致同一个输入框发生位移时不会重新计算位置。
- 根因：只监听 `window.resize`，未监听 `scroll`、局部滚动容器滚动、输入框自身尺寸变化、移动端视口变化。

### B. 与输入框右侧原生按钮重叠
- 现象：图标压住发送/附件/语音等站点内置控件。
- 根因：定位算法仅使用固定 `margin`，未考虑输入框 `padding-right`，也未评估同层右侧控件占位。

### C. 图标出现在“中间偏右”，而非最右侧
- 现象：在部分站点位置明显偏左。
- 根因：定位算法对不同输入框内边距、控件布局、按钮尺寸缺乏自适应，使用固定值计算。

### D. 流式生成期间位置漂移
- 现象：输入框随内容增长（auto-grow）后图标不跟随底部。
- 根因：流式 chunk 写入后未触发重新定位。

## 3. 代码审查发现的其它潜在问题（超出已体感问题）

1. 输入框失效后（DOM 被卸载、SPA 重建）图标可能残留旧位点。
2. 页面存在嵌套滚动容器时，仅监听窗口尺寸变化无法覆盖。
3. iOS/移动端键盘弹出导致 `visualViewport` 变化时无法及时校正。
4. 异步布局变化（动画、延迟渲染）无兜底机制，偶发漂移概率高。

## 4. 本次整改方案

### 4.1 实时跟随机制（事件 + 观察器 + 兜底）
- 去掉“仅按输入框 ID 重定位”的缓存逻辑，改为按需实时重算。
- 新增全局监听：
  - `document scroll (capture)`
  - `window resize`
  - `visualViewport resize/scroll`
- 新增输入框级监听：
  - 监听可滚动祖先容器 `scroll`
  - `ResizeObserver` 监听输入框及一层父容器尺寸变化
- 增加 250ms 兜底同步定时器（处理无事件触发的布局漂移）。
- 通过 `requestAnimationFrame` 合帧节流，避免高频事件造成抖动和性能浪费。

### 4.2 定位算法升级
- 按实际按钮尺寸（CSS 计算值）定位，不再依赖硬编码尺寸。
- 横向定位新增“右侧避让”能力：
  - 读取输入框 `padding-right`
  - 估算同层 sibling 在输入框右半区的控件占位
  - 取最大值作为右侧 inset，降低重叠概率
- 纵向定位保持“单行居中、多行右下”，多行场景额外考虑 `padding-bottom`。
- 增加可见性/连接性检查：目标不可见或已脱离 DOM 时直接隐藏，避免残影。

### 4.3 流式场景修复
- `streamChunk` 写入后主动调度重定位。
- 开始流式清空输入框后调度重定位。
- 流式结束/中断后恢复按钮状态并再次调度定位。

## 5. 实际改动文件

- `src/content/index.ts`
  - 重构按钮显示/隐藏与定位调度逻辑
  - 新增滚动容器收集、ResizeObserver、rAF 调度、兜底同步
  - 接入 `visualViewport` 与流式消息后的重定位触发

- `src/content/ui/button.ts`
  - 升级 `positionButton`：可见性判断、自适应按钮尺寸、padding/右侧控件避让、边界裁剪

## 6. 验证建议（回归清单）

1. ChatGPT/Gemini/Claude/DeepSeek：输入增长、开始生成、停止生成、继续编辑。
2. 包含右侧发送按钮/附件按钮站点：确认不重叠。
3. 页面整体滚动 + 内部滚动容器滚动：确认图标持续贴附。
4. SPA 路由切换前后：确认图标不丢失、不残影。
5. 移动端模拟视口变化（键盘弹出/收起）：确认位置同步。

## 7. 仍需知晓的边界

1. 跨域 iframe 内输入框受浏览器隔离限制，content script 无法直接定位。
2. 极端自定义 Canvas/WebGL 输入 UI 不属于 DOM 可编辑元素，当前策略无法覆盖。
